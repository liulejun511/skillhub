# PR 描述三段式（把侦探活省给每个 reviewer）

## 为何优秀 / 有益
reviewer 的时间最贵。一句话标题 + 空描述，等于把作者省下的几分钟，转嫁给每一个点进来逐行读 diff、来回追问的 reviewer。好的 PR 描述让 reviewer **不看代码就能判断「要不要细看、从哪看起」**，带着上下文进代码而不是当侦探。这条纪律与「最小化改动、按目的分层、证据优先、可讲清」一脉相承：能讲清楚一次改动的背景/范围/验证，本身就是改动质量的体现。

## 任务 (Task)
在提交 PR / MR（或重写一个语焉不详的 PR）时，产出一段让 reviewer 自给自足的描述：背景说清、改动按目的分点、验证给出真实证据。

## 判断重点 (Judgment focus)
- **读者视角**：描述的唯一目标是降低 reviewer 的认知成本——他读完应能决定「细看 / 略过」和「从哪个文件、哪段逻辑看起」。
- **单一目的**：一个 PR 只做一件事。若「改了什么」里冒出多个互不相干的目的，信号是该**拆成多个 PR**，而非把描述写长。
- **证据而非口号**：验证段要贴**真实输出**（计时、行数对账、测试结果、EXPLAIN 前后），不是「本地测过了」这类无法复核的断言。
- **分点而非流水账**：「改了什么」按**目的**归类，不是逐文件罗列 diff（那是 diff 本身的活）。

## 规则 (Rules)
- **标题**：一句话讲清「这个 PR 让什么发生了变化」，避免 `fix bug` / `update` 这类零信息标题。
- **正文三段，缺一不可**：
  1. **为什么改** — 背景与触发的问题；尽量贴上报错信息或现象（复现步骤、错误日志、监控截图）。
  2. **改了什么** — 按目的分点说明；混入多目的就拆 PR。
  3. **怎么验证的** — 跑了什么、贴关键输出（耗时前后对比、测试通过数、行数核对、EXPLAIN 计划差异）。
- 可直接复制的模板：

```markdown
## 为什么改
<背景 / 触发的问题；贴报错或现象>

## 改了什么
- <目的一：做了什么>
- <目的二：做了什么>

## 怎么验证的
- <跑了什么命令 / 测试>
- <关键输出，如 EXPLAIN 前后耗时、测试结果、行数对账>
```

- 描述里出现「顺手还改了 X」时，停下来问：X 是否属于本 PR 的单一目的？不属于就拆出去。

## 不适用 (Not for)
- 纯机械、零风险的改动（typo、格式化、版本号 bump）——一句话标题足矣，别套三段式徒增噪音。
- 不是用来审查 diff **内容**正确性的（那是 code review / 逐行质询的活，见 [[ai-diff-interrogation]]）；本技能只管「把改动讲清楚给人看」。
- 不替代提交前的最小化改动与分层提交纪律——描述写得好，不能洗白一个该拆的大杂烩 PR。

## 输出 / 行动结构 (Output / Action)
按上面的模板产出标题 + 三段描述：
1. 标题：一句话点明变化。
2. 为什么改：背景 + 现象/报错。
3. 改了什么：按目的分点（多目的 → 拆 PR）。
4. 怎么验证的：真实证据输出。
自检：reviewer 只读这段，能否判断「细看/略过」与「从哪看起」？不能则补足。
