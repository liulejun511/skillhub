# 证据先行，验真不验吹

## 为何优秀 / 有益
这是用户自身已被验证的强项做法，值得固化：任何「更快了/修好了/自动化了」的主张，在拿到**真实数据证据**之前都只是叙事。坚持「你给我命令我跑一下，没问题就改」，用 EXPLAIN ANALYZE、计时、行数对账说话；对 AI/同事的工作汇报同样追问「哪些是真的、谁做的」——能避免被乐观叙事带进坑，也让每次决策可复盘。

## 任务 (Task)
在接受一个改动、一个性能主张或一份工作汇报之前，设计并执行最小验证，拿到证据再放行。

## 判断重点 (Judgment focus)
- **这个主张的可证伪验证是什么？** SQL 改动 → EXPLAIN (ANALYZE, BUFFERS)；接口 → curl 计时多次取样；迁移 → 前后行数/抽样对账；修 bug → 原始触发路径复现。
- **环境对不对？** 在影子/预发环境验证，别拿生产冒险；确认连的是哪个库、哪个分支。
- **汇报与事实的差距**：自动化汇报里哪些是工具真跑的、哪些是人肉代劳的？「跑通了」是端到端跑通还是只跑了一段？
- **单点 vs 重复**：一次成功不算数，重复取样（如连发 5 次看耗时分布）。

## 规则 (Rules)
- 没证据不合并、不上线、不下结论；"感觉快了" 不是证据。
- 验证命令自己跑或亲眼看到输出，不接受转述。
- 验证发现意外（比如比预期慢/卡住），先定位再继续，不硬推。
- 给出证据时附上下文：环境、数据量、命令本身，让结论可复查。

## 回归取证（行为退化专用，v0.2 强化）
当「以前是好的，现在坏了」时，多两道取证：
- **A/B 复现**：验证修复前，**先在未修复版本上把故障复现出来**——坏版本复现不了，「修好了」就不可证伪。先测原版、再测修复，对照才有效。
- **时间线取证**：别用记忆争论「上周还好好的」。用硬时间戳锚定变更窗口：容器 `Created`、基础设施创建时间、git log。
```bash
docker inspect <容器> --format '{{.Created}}'          # 这个版本啥时候起的
aws elasticache describe-serverless-caches --query '...CreateTime'  # 依赖啥时候换的
git log --since="2 weeks ago" --oneline -- <相关路径>   # 这段时间动了什么
```

## 不适用 (Not for)
- **全新功能**（没有「以前」可对照）：用常规验收，不必做 A/B 复现。
- **探索/原型阶段**：快速试错时过度取证会拖慢迭代，证据要求随接近上线而提高。
- **纯主观偏好**（措辞、配色）：没有可证伪指标的事别硬套证据流程。

## 输出 / 行动结构 (Output / Action)
1. **主张** — 要验证的具体说法。
2. **验证设计** — 命令/步骤 + 在哪个环境；退化类问题加 A/B 复现与时间线锚点。
3. **证据** — 实际输出（计时/计划/行数/时间戳）。
4. **判定** — 接受 / 退回 / 需要进一步定位，及理由。
